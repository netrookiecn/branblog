所有人都知道多模态 LLM 要"看得懂图片"，但我原来以为：视觉信息喂得越精细，模型理解越好。

翻完智谱这篇 GLM-5V-Turbo tech report，我的判断变了：**把视觉 embedding 直接塞给 LLM 可能是个误区——用简单的 `<|image|>` 占位符反而效果更好**。

这不是玄学。论文在 0.5B 模型上做了 ablation：直接传 visual embedding 到 MTP head，loss 更高、收敛更不稳定；用 `<|image|>` 反而更低。这背后的原因是 MTP head 通常很轻量，可能没有足够的建模能力来吸收和文本分布差异巨大的视觉表示。

这篇文章我想讲三件事：
1. **原生多模态 Agent 是什么**——不是"视觉语言模型 + 工具"，而是"感知即推理"
2. **MMTP 的设计权衡**——为什么简单占位符比复杂 embedding 更好
3. **三个设计 lens**——从实践中归纳出的多模态 Agent 训练方法论

---

## 一、共识：多模态 Agent 的范式转移

过去两年，大家训练 VLM（Vision-Language Model）的思路是：**视觉编码器很强 + 语言模型很强 + 简单的 projection 层把它们对齐**。

这个范式把视觉当成"辅助输入"——图片先被 vision encoder 打成 token，然后和文本 token 一起喂给 LLM。模型的任务还是"理解图片后回答问题"。

GLM-5V-Turbo 提出的范式不同：**多模态感知是推理、规划、工具使用、执行的核心组件，不是辅助接口**。

换句话说，Agent 需要在视觉上下文中"思考"和"行动"，而不是"看图 + 说话"。

> 这一节讲的是"大家都接受的旧范式"；下一节看新范式带来什么具体设计挑战。

## 二、分歧：CogViT、MMTP、与层级优化

### 2.1 CogViT 视觉编码器

智谱自研的 CogViT 是一个参数高效的视觉编码器，在通用目标识别、细粒度理解、几何和空间感知上都达到 SOTA 水平（Figure 1）。

**两阶段预训练：**

**Stage 1：蒸馏式 MIM（掩码图像建模）**

- 用 SigLIP2（语义表示）和 DINOv3（纹理特征）作为 teacher
- 学生 ViT 重建被 mask 的区域（35% mask ratio, 224×224）
- 数据配比：80% 高质量自然图像 + 10% 指令跟随数据 + 10% 科学图像
- 用 Muon optimizer + cosine decay

**Stage 2：对比学习预训练**

- 替换固定分辨率，用 NaFlex 方案处理可变大小输入同时保持原始宽高比
- 全局 batch size 扩大到 64K，使用基于 sigmoid 的 SigLIP loss
- 用 80 亿双语（中文-英文）图像-文本对增强跨语言理解

关键设计：**引入 QK-Normalization**，在 attention 计算前对 query 和 key 向量做归一化，有效缓解 logit 爆炸问题，保证大规模训练的稳定性。

### 2.2 多模态多 Token 预测（MMTP）

这是我认为最有趣的技术决策。

标准的多 token 预测（MTP）在文本上很有效——预测多个未来 token 而不是只预测下一个。但扩展到多模态输入时，一个核心问题出现了：**图像 token 应该怎么传给 MTP head？**

论文系统比较了三种方案：

![MMTP Three Design Options](/assets/images/02-mmtp-comparison.svg)

| 方案 | 描述 | 问题 |
|------|------|------|
| **Option 1** | 直接把 visual embedding 传给 MTP head | 训练 loss 高、收敛不稳定 |
| **Option 2** | 在 MTP head 输入时 mask 掉所有 visual token | 退化为纯文本 MTP |
| **Option 3（采用）** | 用一个共享的可学习 `<|image|>` token 作为视觉输入表示 | 简单有效 |

为什么简单反而更好？论文的分析：

1. **MTP head 通常很轻量**，没有足够的建模能力来吸收和文本分布差异巨大的视觉表示
2. **`<|image|>` token 以更统一的形式呈现输入**，缓解了优化困难
3. **兼容现有的序列并行和上下文并行**，不需要额外的 visual embedding 分区处理、对齐或 offset mapping

实测结果（0.5B 模型 ablation）：Option 3 的训练 loss 更低、收敛更稳定。

> 这一节讲的是技术方案选择；下一节看更宏观的训练策略。

### 2.3 广泛的多模态 RL 训练

GLM-5V-Turbo 在超过 **30 个任务类别** 上做联合 RL 优化。这不是"在某个 benchmark 上刷分"，而是覆盖感知、推理、Agent 能力的全面训练。

**感知侧提升：**
- 2D 图像定位和指向：+4.8% on RefCOCO-avg, +3.2% on PointBench
- 视频理解：+5.6% on MVBench
- 3D 定位：+7.7% on SUNRGBD
- OCR：+4.2% on OCRBench
- 图表理解：+7.7% on CharXiv

**推理侧提升：**
- STEM 任务：+1.8% on MMMU_Val / MMMU_Pro / MathVista / LogicVista
- 问题解决更稳定

**Agent 侧提升：**
- GUI Agent：+4.9% on OSWorld
- 编码 Agent：+0.2% on CC-Backend
- 通用工具使用：+3.5% on MMSearch（规划和执行能力提升）

**一个有趣的发现：**
多任务 RL 的跨域干扰比 SFT 更弱，多个领域可以一起提升，而且分布较窄的领域通过协作训练变得更稳定。更重要的是，**推理行为可以在任务间迁移**——在一个领域学到的推理模式可以迁移到其他领域并产生可测量的收益。

> 这一节讲的是具体技术；下一节看实际效果和 benchmark。

## 三、暗线：三个设计 Lens

翻完整篇 report，我归纳出智谱在多模态 Agent 开发中反复验证的三个设计视角：

### Lens 1：感知仍是高层能力的基础

> "Many failures that appear high-level, in other words, begin with the model not seeing the environment accurately enough."

最近的讨论都强调规划、推理、反思等"高层"能力。但实际观察表明，**多模态能力的进一步提升仍然关键依赖于感知**。即使是最强的当前 VLMs，细粒度感知和空间理解的错误仍然常见，这些错误往往会传播到下游推理和决策。

**有价值的代理任务：**
- 多模态编码和定位是有用的感知学习代理任务
- 前端或 SVG 编码要求模型捕捉布局、结构、相对位置和局部细节，不只是粗粒度语义
- 在预训练时加入配对的 subject-specific images 和 SVG 表示，能正向贡献下游 STEM 问题解决
- 在 RL 时加强定位相关训练也能改善 GUI Agent 性能

**显式训练自我批评：**
- 在 GUI Agent 的指令调优中加入 critic 数据，目标是推理过程中的错误
- 这改善了 GUI 细节的观察质量，减少了几种持续的感知失败模式

### Lens 2：层级优化比端到端训练更有效

Agent 训练本身资源密集：环境设置和任务构建成本高，高质量数据稀缺，可靠验证往往困难。同时，Agent 任务本身难以高效优化，因为它们通常是长轨迹、多步骤的。

这导致智谱采用**层级优化策略**：
- 不是只在高级长程任务上集中优化
- 而是跨越能力层级（元素感知 → GUI 定位 → 单步动作预测 → 轨迹级动作预测）的多个级别上分布优化
- 在 SFT 和 RL 中都使用多级任务层级

这个设计的吸引力是双重的：
- 较低级别的任务在相同资源约束下通常更容易构建、标注和验证
- 当较低级别能力仍然不成熟时，只在高级任务上推动往往无法产生可靠收益，反而可能使训练更不稳定

### Lens 3：任务规范、验证、评估的Procedural Control

> "The real challenge is often not extending tasks to longer horizons, but making end-to-end tasks stable enough to serve as meaningful targets for evaluation and optimization."

对于多模态 Agent，真实的挑战往往不是把任务延伸到更长的视野，而是**让端到端任务足够稳定**，能够作为评估和优化的有意义的目标。

许多现实的 Agent 设置本质上是开放式的：目标不明确、执行边界模糊、结果严重依赖中间决策。结果是它们往往难以一致比较，甚至难以转化为可重用的优化信号。

这催生了一个更宽广的视角：
- 一个端到端任务的价值不仅取决于它有多"现实"
- 还取决于它是否能够足够清晰地指定、足够可靠地验证、足够在程序控制下评估，以产生稳定和可重用的反馈

**Vision2Web benchmark** 是这个视角的具体实例：
- 每个任务不仅基于文本指令，还基于更丰富的规范（PRD、mockup、参考页面、资源资产）
- 在评估方面，不是把网站开发当作一个松散指定的开放式问题，而是使用基于工作流的验证
- 执行通过受控的依赖步骤序列来评估，而不是单一最终状态

### 我现在的选择

如果我明天要做多模态 Agent：

1. **会用 `<|image|>` token 而不是直接传 visual embedding**，特别是在 MTP 场景下
2. **会采用层级优化而不是端到端训练**，从感知 → 定位 → 单步 → 轨迹逐步构建
3. **会把任务规范和验证设计放在和模型训练同等重要的位置**，因为"真实感"不等于"可优化"

### 待验证的问题

1. **`<|image|>` token 在更大模型上的效果**：0.5B 上的结论能推广到 10B+ 吗？
2. **层级优化在动态环境（ 比如手机/桌面自动化）上的效果**：任务层级如何设计？
3. **长程多模态记忆机制**：图片/视频比文本消耗 context 预算更激进，如何压缩保留视觉细节？
