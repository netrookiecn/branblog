所有人都知道多模态 LLM 要"看得懂图片"，但我原来以为：视觉信息喂得越精细，模型理解越好。

翻完智谱这篇 GLM-5V-Turbo tech report，我的判断变了：**把视觉 embedding 直接塞给 LLM 可能是个误区——用简单的 `<|image|>` 占位符反而效果更好**。

这不是玄学。论文在 0.5B 模型上做了 ablation：直接传 visual embedding 到 MTP head，loss 更高、收敛更不稳定；用 `<|image|>` 反而更低。这背后的原因是 MTP head 通常很轻量，可能没有足够的建模能力来吸收和文本分布差异巨大的视觉表示。

这篇文章我想讲三件事：
1. **原生多模态 Agent 是什么**——不是"视觉语言模型 + 工具"，而是"感知即推理"
2. **MMTP 的设计权衡**——为什么简单占位符比复杂 embedding 更好
3. **三个设计 lens**——从实践中归纳出的多模态 Agent 训练方法论

---

## 一、共识：多模态 Agent 的范式转移

过去两年，大家训练 VLM（Vision-Language Model）的思路是：**视觉编码器很强 + 语言模型很强 + 简单的 projection 层把它们对齐**。

这个范式把视觉当成"辅助输入"——图片先被 vision encoder 打成 token，然后和文本 token 一起喂给 LLM。模型的任务还是"理解图片后回答问题"。

GLM-5V-Turbo 提出的范式不同：**多模态感知是推理、规划、工具使用、执行的核心组件，不是辅助接口**。换句话说，Agent 需要在视觉上下文中"思考"和"行动"，而不是"看图 + 说话"。

![旧范式 vs 原生多模态 Agent 范式](/assets/images/03-paradigm-shift.svg)

> 这一节讲的是"大家都接受的旧范式"；下一节看新范式带来什么具体设计挑战。

## 二、分歧：CogViT、MMTP、与层级优化

### 2.1 CogViT 视觉编码器

智谱自研的 CogViT 是一个参数高效的视觉编码器，在通用目标识别、细粒度理解、几何和空间感知上都有竞争力的表现（Figure 1）。

两阶段预训练的核心思路是：**先通过蒸馏建立强视觉表示，再通过对比学习做视觉-语言对齐**。Stage 1 用 SigLIP2（语义）+ DINOv3（纹理）作为双教师做蒸馏式 MIM；Stage 2 引入 NaFlex 可变分辨率、SigLIP loss 和 8B 双语语料做对比学习预训练。关键工程细节是 QK-Normalization——在 attention 计算前对 query 和 key 向量归一化，防止 logit 爆炸。

![CogViT 两阶段预训练流程](/assets/images/04-cogvit-training.svg)

### 2.2 多模态多 Token 预测（MMTP）

这是我认为最有趣的技术决策。标准的多 token 预测（MTP）在文本上很有效——预测多个未来 token 而不是只预测下一个。但扩展到多模态输入时，一个核心问题出现了：**图像 token 应该怎么传给 MTP head？**

论文系统比较了三种方案，最终选择了看起来最简单的 Option 3——用一个共享可学习的 `<|image|>` token 代替所有视觉位置的输入。Option 1（直接传 visual embedding）训练 loss 更高，收敛不稳定；Option 2（mask 掉所有视觉 token）则退化为纯文本 MTP，失去了多模态能力。

![MMTP 三种设计方案对比](/assets/images/02-mmtp-comparison.svg)

为什么 `<|image|>` 更好？核心原因有三：MTP head 通常很轻量，建模能力不足以吸收分布差异巨大的视觉 embedding；`<|image|>` 以更统一的形式呈现输入，缓解了优化困难；同时完全兼容现有的序列并行和上下文并行分区策略，不需要额外的 visual embedding 分区处理和 offset mapping。

> 这一节讲的是技术方案选择；下一节看更宏观的训练策略。

### 2.3 广泛的多模态 RL 训练

GLM-5V-Turbo 在超过 **30 个任务类别** 上做联合 RL 优化——覆盖感知、推理、Agent 能力的全面训练，而不是在某个 benchmark 上单点刷分。

论文报告的提升横跨三个层面：感知侧（RefCOCO-avg +4.8%，SUNRGBD +7.7%，CharXiv +7.7%），推理侧（MMMU 系列 +1.8%），Agent 侧（OSWorld +4.9%，MMSearch +3.5%）。一个值得关注的发现：多任务 RL 的跨域干扰比 SFT 更弱，**推理模式可以在任务间迁移**——在一个领域学到的策略可以正向影响另一个领域。

![多模态 RL 30+ 任务覆盖的提升幅度](/assets/images/05-rl-coverage.svg)

## 三、暗线：三个设计 Lens

翻完整篇 report，我归纳出智谱在多模态 Agent 开发中反复验证的三个设计视角，这三个 lens 都不是"公认的正确做法"，而是在实践中踩过坑之后摸出来的。

### Lens 1：感知仍是高层能力的基础

> "Many failures that appear high-level, in other words, begin with the model not seeing the environment accurately enough."（§4, Lens 1）

最近的讨论都在强调规划、推理、反思等"高层"能力，但论文的观察是：细粒度感知和空间理解的错误仍然常见，而且这些错误会传播到下游的推理和决策——许多表现为"高层能力不足"的失败，根源其实在感知层看不准。

![感知错误的传播链](/assets/images/06-perception-chain.svg)

解法不是只训"感知"，而是用多模态编码、定位等结构化代理任务来强化感知基础：SVG/前端编码要求模型捕捉布局和空间关系；预训练时加入 subject-specific images 和 SVG 配对数据，能正向影响 STEM 问题解决；RL 时加强定位训练也能改善 GUI Agent 性能。另外，在 GUI Agent 的指令调优中加入显式 critic 数据（针对推理错误的自我批评），也有效减少了感知失败模式的复发。

### Lens 2：层级优化比端到端训练更有效

Agent 训练本质上是资源密集的：环境搭建成本高，高质量轨迹数据稀缺，可靠的结果验证困难。在这个约束下直接推端到端的长程任务，低层能力不够时信号极其稀疏，训练容易震荡。

智谱采用了**层级优化策略**：构建从"元素感知 → GUI 定位 → 单步动作 → 轨迹级任务"的四级能力层级，在 SFT 和 RL 中都沿这个层级分布优化。好处是双重的：底层任务更容易构建标注和验证，同时底层能力积累扎实后，高层任务的优化信号才变得有效。

![层级优化 vs 端到端训练](/assets/images/07-hierarchical-opt.svg)

### Lens 3：任务规范、验证、评估要联合设计

> "The real challenge is often not extending tasks to longer horizons, but making end-to-end tasks stable enough to serve as meaningful targets for evaluation and optimization."（§4, Lens 3）

对多模态 Agent 来说，很多现实 Agent 设置本质上是开放式的——目标不明确、执行边界模糊、结果高度依赖中间决策——这使得它们难以一致比较，更难转化为可重用的优化信号。

论文的视角是：一个端到端任务的价值，不只取决于它有多"真实"，还取决于它是否能**足够清晰地指定、足够可靠地验证、足够在程序控制下评估**，以产生稳定反馈。Vision2Web benchmark 是这个视角的具体实例：任务规范包含 PRD + mockup + 参考页面 + 资源资产，验证采用工作流式依赖步骤序列而不是单一最终状态，评估分离功能正确性和视觉一致性两个维度。

![任务规范 × 验证设计 × 评估控制](/assets/images/08-task-spec-verify.svg)

## 四、效果与我的判断

### Benchmark 结果

GLM-5V-Turbo 在多模态编码、工具使用、GUI Agent、文本编码四个方向都有实测数据。最值得关注的是 Design2Code 94.8（论文称超过 Claude Opus 4.6）和 AndroidWorld 75.7，以及文本编码方向维持了和语言基础模型 GLM-5-Turbo 相当的水平——视觉能力没有拖累文本能力。

![GLM-5V-Turbo Benchmark 横向对比](/assets/images/09-benchmark-results.svg)

### 我现在的选择

![多模态 Agent 技术选型决策](/assets/images/10-decision-summary.svg)

如果明天要做多模态 Agent：

1. **会用 `<|image|>` token 而不是直接传 visual embedding**，特别是在 MTP 场景下——0.5B 的 ablation 结果很清楚
2. **会采用层级优化而不是端到端训练**，从感知 → 定位 → 单步 → 轨迹逐步构建，底层扎实再推高层
3. **会把任务规范和验证设计放在和模型训练同等重要的位置**，因为"真实感"不等于"可优化"

### 待验证的问题

![待验证的技术问题](/assets/images/11-open-questions.svg)

1. **`<|image|>` token 在更大模型上的效果**：0.5B 上的结论能推广到 10B+ 吗？MTP head 更大时结论还成立吗？
2. **层级优化的设计成本**：任务层级怎么划分？不同任务间的比例如何调配？
3. **长程多模态记忆机制**：图片/视频消耗 context 预算远比文本激进，如何在压缩时保留关键视觉细节？
